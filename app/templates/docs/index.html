Rate Limiting

API requests are rate-limited using a sliding window algorithm. When you exceed the limit, you'll receive a 429 response with a Retry-After header indicating how long to wait. The X-RateLimit-Remaining response header shows your remaining quota.

How It Works

Request flow

When an AI agent (like Claude Code) tries to use a tool, the hook intercepts the request and sends it to Snapper for evaluation before it executes:

1 Agent wants to run rm -rf /tmp/test
↓
2 Hook calls POST /api/v1/rules/evaluate with the tool name and input
↓
3 Snapper checks the request against all active rules (highest priority first)
↓
4 Snapper returns a decision — the hook acts on it

Decisions explained

Every evaluate call returns one of three decisions. Here's what each one means for the agent:

allow Action proceeds normally

The hook returns an empty response (exit code 0) and the agent's tool call executes as if Snapper wasn't there. This happens when the request matches an allow rule, or when no deny rules match.

deny Action is blocked immediately

The hook returns an error message and the agent sees "Snapper blocked this action: <reason>". The tool call never executes. The agent can see the reason and may try an alternative approach. The block is logged in the Audit page.

require_approval Action pauses and waits for a human

The hook pauses the agent and polls GET /api/v1/approvals/{id}/status every few seconds. You get a notification (via Telegram or the dashboard) to approve or deny. If you approve, the tool call executes. If you deny or it times out (default: 5 minutes), the action is blocked.

What triggers a block?

Blocks come from your active rules. Common reasons you'll see in the audit log:

Rule Type What It Blocks Example Block Reason
command_denylist Shell commands matching denied patterns "Command matches denied pattern: rm -rf"
credential_protection Access to sensitive files (.env, keys, certs) "File matches protected pattern: .env"
skill_denylist Specific MCP tools flagged as malicious "Skill blocked: malicious-skill-name"
rate_limit Too many requests in a time window "Rate limit exceeded: 10/60s"
time_restriction Actions outside allowed hours "Outside allowed hours: 09:00-17:00"
human_approval Sensitive actions (returns require_approval) "Requires human approval for: pip install"

If no rules match a request, Snapper denies by default (fail-safe). To allow an action, create an explicit allow rule. See Creating Rules for details.

Rules API

Create, manage, and evaluate security rules that control agent behavior.

POST /api/v1/rules/evaluate Evaluate a request against rules

The primary hook integration point. The hook script calls this endpoint every time the agent tries to use a tool. Snapper evaluates the request against all active rules and returns a decision (allow, deny, or require_approval). See How It Works for what each decision does.

Example request:

curl -X POST http://localhost:8000/api/v1/rules/evaluate \
  -H "Content-Type: application/json" \
  -d '{
    "agent_id": "claude-code-macbook",
    "tool_name": "Bash",
    "tool_input": {
      "command": "rm -rf /tmp/test"
    }
  }'

Example response:

{
  "decision": "deny",
  "rule_id": "rule_abc123",
  "rule_name": "Dangerous Commands Denylist",
  "reason": "Command matches denied pattern: rm -rf"
}

Possible decisions: allow, deny, require_approval

GET /api/v1/rules List all rules

Retrieve rules with optional filtering by agent, type, and active status.

Example request:

curl "http://localhost:8000/api/v1/rules?agent_id=claude-code-macbook&is_active=true&skip=0&limit=20"

Example response:

[
  {
    "id": "rule_abc123",
    "name": "Credential Protection",
    "agent_id": "claude-code-macbook",
    "rule_type": "credential_protection",
    "parameters": {
      "patterns": [".env", "*.pem", "id_rsa"]
    },
    "is_active": true,
    "priority": 100
  }
]

Query params: agent_id, rule_type, is_active, skip, limit

POST /api/v1/rules Create a new rule

Create a rule to control agent behavior. Rules are evaluated in priority order (highest first).

Example request:

curl -X POST http://localhost:8000/api/v1/rules \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Block env files",
    "agent_id": "claude-code-macbook",
    "rule_type": "credential_protection",
    "parameters": {
      "patterns": [".env", ".env.*", "*.pem"]
    },
    "is_active": true,
    "priority": 100
  }'
GET /api/v1/rules/{rule_id} Get rule by ID 
curl http://localhost:8000/api/v1/rules/rule_abc123
PUT /api/v1/rules/{rule_id} Update a rule 
curl -X PUT http://localhost:8000/api/v1/rules/rule_abc123 \
  -H "Content-Type: application/json" \
  -d '{
    "is_active": false,
    "priority": 50
  }'
DELETE /api/v1/rules/{rule_id} Delete a rule

Soft-deletes a rule by default. Add ?permanent=true for hard delete.

curl -X DELETE http://localhost:8000/api/v1/rules/rule_abc123
POST /api/v1/rules/validate Validate a rule (dry run)

Validate a rule definition without creating it. Returns validation errors if any.

curl -X POST http://localhost:8000/api/v1/rules/validate \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Test rule",
    "rule_type": "command_denylist",
    "parameters": { "commands": ["rm -rf /"] }
  }'
GET /api/v1/rules/templates List rule templates

Get pre-built rule templates for common security scenarios.

curl http://localhost:8000/api/v1/rules/templates
POST /api/v1/rules/templates/{template_id}/apply Apply a rule template 
curl -X POST http://localhost:8000/api/v1/rules/templates/credential-protection/apply \
  -H "Content-Type: application/json" \
  -d '{ "agent_id": "claude-code-macbook" }'
POST /api/v1/rules/export Export rules

Export rules as JSON or YAML for backup or sharing.

curl -X POST http://localhost:8000/api/v1/rules/export \
  -H "Content-Type: application/json" \
  -d '{ "format": "json", "agent_id": "claude-code-macbook" }'
POST /api/v1/rules/import Import rules

Import rules from a JSON payload.

curl -X POST http://localhost:8000/api/v1/rules/import \
  -H "Content-Type: application/json" \
  -d '{ "rules": [ ... ] }'

Example request:

Error Handling

All error responses follow a consistent JSON format with a detail field.

Code Meaning Common Cause
400 Bad Request Invalid JSON or missing required fields
404 Not Found Rule, agent, or resource does not exist
409 Conflict Resource already exists (e.g., duplicate agent ID)
422 Validation Error Request body fails schema validation
429 Too Many Requests Rate limit exceeded; check Retry-After header
500 Internal Error Unexpected server error

Example error response:

{
  "detail": "Rule not found: rule_abc123"
}

Validation error response (422):

{
  "detail": [
    {
      "loc": ["body", "rule_type"],
      "msg": "field required",
      "type": "value_error.missing"
    }
  ]
}
