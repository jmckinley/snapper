Data Protection

PII Vault

Encrypted storage with zero-knowledge token resolution

1. Store
User adds PII via Telegram / Slack / API ↓
2. Encrypt
AES-256-GCM, HKDF-derived key ↓
3. Tokenize
{{SNAPPER_VAULT:a1b2...}} (128-bit) ↓
4. Approve
Human reviews destination + data fields ↓
5. Resolve
Decrypted value returned (30s TTL)

Security Properties

  • AES-256-GCM (authenticated encryption)
  • HKDF key derivation from SECRET_KEY
  • 128-bit token entropy
  • 30-second TTL on resolved values
  • Brute-force lockout (5 failures/15min)
  • Per-user encryption isolation

Features

  • Domain whitelisting
  • Multi-tenant (owner scoped)
  • Manage via bot, API, or dashboard
PII never appears in agent context or logs. Only the token moves through the system. Resolution happens at the last moment with human approval. 30-second TTL means decrypted values don't linger.

Security Design

Security Architecture

6 Design Principles

  • 1. Deny by default — no rule = blocked
  • 2. DENY always short-circuits
  • 3. Priority-based evaluation
  • 4. No rule caching — always fresh
  • 5. Learning mode for safe onboarding
  • 6. Errors = deny (fail-closed)

6 Enforcement Layers

Request ↓ L1: Security Middleware (Host/Origin/CORS) ↓ L2: API Key Authentication ↓ L3: Rule Engine (16 types, priority-sorted) ↓ L4: PII Gate (vault tokens + raw scan) ↓ L5: Approval Workflow (Telegram/Slack) ↓ L6: Audit Trail (immutable, BRIN-indexed)

Each layer is independent — failure in any layer defaults to deny  |  No single point of bypass

Defense in depth. Six independent layers mean an attacker would need to bypass all of them. Fail-closed design means any error results in deny, not pass.
